## Script: `Migration-OD2OD-SPO(09132024).ps1`

**Purpose:** Migrate on-prem OneDrive (MySite) content into a user's SPO
OneDrive.

**Source pattern:** `https://onedrive.contoso-onprem.local/my/<sam>`
**Target pattern:** `https://contoso-my.spo.microsoft.scloud/personal/<upn-with-underscores>`

**Driver list:** `OneDriveMigrationStatus` on
`https://contoso.spo.microsoft.scloud/sites/000001`

**Key parameters (read from the SPO list):**
- `Title` — display name / SAM
- `SourceUrl` — on-prem MySite URL
- `TargetUrl` — SPO OneDrive URL
- `Migrate` — set to literal value `"Migrate"` to trigger processing (NOT
  `"Ready"` — the script's filter is `$item["migrate"] -eq "Migrate"`).
  Terminal values written by the script: `Migrated`, `ErrorLog`, `Failed`.
- `Inprocess` — **lock flag for OD2OD** (set to `Processing` while running,
  cleared on completion). Note: SP2SPO uses a different column name `Processing`.
- `Log` — populated with the log path on success (not `MigrationLog`)
- `StartDate`, `CompletedDate` — timestamps

**Sleep between users:** 30 seconds.

**DLL load path:** `F:\Tools\` (loaded via `Add-Type` before connecting).

**Credentials prompted at start:**
- SPO admin (UPN/Pass)
- On-prem SharePoint admin

**Post-migration side effects (on success):**
- **SCA swap on the SOURCE on-prem MySite** (`$sourceUrl`, NOT the new SPO
  OneDrive):
  1. Ensure `svc-migration@contoso.gov` exists on the source site.
  2. Make `svc-migration` a Site Collection Administrator on the source.
  3. Demote the migrated user from SCA on the source.
  This locks down the source MySite so users can't keep editing it.
- **AD updates RUN by default.** The script DOES:
  - Set the user's AD `wwwHomePage` attribute to the new SPO OneDrive URL.
  - Remove the user from all groups matching `*REDIRECTION*` and from
    `SecFltr-USR-OneDrive`.
  - Add the user to `SecFltr-USR-Office365`.
- **⚠️ Comment-marker caveat:** the script has `#<#` ... `#>` markers around
  the AD block that LOOK like a block comment, but PowerShell parses
  `#<#` as a single-line comment (the leading `#` makes the whole line a
  comment), so `<#` never opens a block. The AD code runs. To truly
  disable AD changes, change `#<#` to `<#` (remove the leading `#`) on the
  marker lines.
- **FailureSummaryReport.csv** (if generated by SPMT) is attached to the
  list item and `Migrate` is set to `ErrorLog`. **The script does NOT send
  email** — a Power Automate flow that watches the list is what notifies
  the user.

**Author/version:** Doug Cox, 03 June 2024, v3.2

## Script: `Migration-SP2SPO(09132024).ps1`

**Purpose:** Migrate on-prem SharePoint sites into SPO sites.

**Source pattern:** any on-prem SP site URL (SP 2016 / 2019).
**Target pattern:** `https://contoso.spo.microsoft.scloud/sites/<sitename>`

**Driver list:** `SPOMigrationStatus` on
`https://contoso.spo.microsoft.scloud/sites/000001`

**Key SPMT call:**
```powershell
Register-SPMTMigration -SPOCredential $SPOCredential `
    -SkipFilesWithExtension $BlockedExtensions `
    -WorkingFolder $LoggingSharePointMigration `
    -KeepAllVersions $true `
    -MigrateFileVersionHistory $true `
    -ReplacementOfInvalidChar "_"

Add-SPMTTask -SharePointSourceSiteUrl $site `
    -TargetSiteUrl $targetUrl -MigrateAll `
    -SharePointSourceCredential $OnPremCredential
```

**Lock-flag column:** `Processing` (set to `Processing` while running). Note
this is DIFFERENT from OD2OD, which uses `Inprocess`.

**Sleep between sites:** 120 seconds.

**DLL load path:** `F:\IAU_Scripts\OneDrive_Migration_Scripts\`.

**Status values written to `Migrate`:** `Ready` -> `Migrated` (success) /
`ErrorLog` (per-file errors, CSV attached) / `Failed` (hard fail).

**Power Automate:** the script does not email. A flow on the list watches
for `ErrorLog` / `Failed` and sends the email.

**Author/version:** Doug Cox, 03 June 2024, v3.2

## OD2OD vs SP2SPO — quick comparison

| Aspect | OD2OD (Flow A) | SP2SPO (Flow B) |
|---|---|---|
| Source | On-prem MySite | On-prem SP site |
| Target | SPO OneDrive `/personal/` | SPO site `/sites/` |
| Driver list | `OneDriveMigrationStatus` | `SPOMigrationStatus` |
| Lock-flag column | `Inprocess` | `Processing` |
| SPMT add-task | Per-user MySite | Per-site, `-MigrateAll` |
| Sleep between items | **30 seconds** | **120 seconds** |
| DLL load path | `F:\Tools\` | `F:\IAU_Scripts\OneDrive_Migration_Scripts\` |
| SCA swap post-migration | Yes (svc-migration) on SOURCE MySite ($sourceUrl) | Yes |
| AD updates | wwwHomePage + group flips (run by default; see comment-marker caveat above) | Site owner / group flips (similar caveats apply) |
| On-prem cleanup | Remove user's MySite perms | Remove site perms |
| Notification | Power Automate flow emails user on `ErrorLog`/`Failed` | Power Automate flow emails owner |

## Prerequisites (both flows)

- SPMT 4.2.129.0+ installed at:
  `F:\SPMT-migration_tool.9\microsoft.sharepoint.migrationtool.advancedapp.exe`
- PowerShell modules: SPMT, SharePointPnPPowerShellOnline (or PnP.PowerShell),
  PnP DLLs from the SPMT folder
- Two credentials available (will be prompted via `Get-Credential`):
  - SPO admin (UPN `*@contoso.gov`)
  - On-prem SharePoint admin
- Driver SPO list provisioned with required columns
- Power Automate flow on the driver list (optional, for error emails)
- Network line of sight: migration host → on-prem SP farm

## Common errors

| Symptom | Likely cause | Fix |
|---|---|---|
| `Cannot find SPMT module` | SPMT not installed / wrong path | Install SPMT, verify `$SPMTPATH` |
| `Add-Type` DLL load failure | Wrong DLL path (OD2OD vs SP2SPO use different folders) | Verify `F:\Tools\` (OD2OD) or `F:\IAU_Scripts\OneDrive_Migration_Scripts\` (SP2SPO) |
| `Access denied` on source | On-prem cred wrong or missing perms | Re-prompt `$OnPremCredential` |
| Lock flag stuck after crash | Crashed mid-run | Reset the lock column to blank in the list. **OD2OD column is `Inprocess`, SP2SPO column is `Processing`** — don't confuse them. |
| `FailureSummaryReport.csv` attached + `Migrate = ErrorLog` | Per-file errors (long paths, invalid chars, blocked extensions) | Review the CSV from the list item attachments, re-run for those items |
| User got no email after a failure | Power Automate flow not configured — the script never emails | Build the flow on the list (trigger: `Migrate` = `ErrorLog`/`Failed`) |
| AD update didn't happen | Likely an AD permission error on the runner account, OR someone replaced `#<#` with `<#` to truly disable the block | Check transcript for AD-related errors; verify the runner account has `Remove-ADGroupMember` / `Set-ADUser` rights; check the script's `#<#` marker hasn't been edited to `<#` |
| Some file types not migrated | ~50 blocked extensions are skipped by design (`.pst`, `.tmp`, `.lnk`, etc.) | Edit `$BlockedExtensions` in the script header to change |

## Sample invocation

```powershell
# Both scripts are self-contained and prompt for credentials.
# Run from the migration host:

# Flow A — MySites
.\Migration-OD2OD-SPO(09132024).ps1

# Flow B — Sites
.\Migration-SP2SPO(09132024).ps1
```

The script reads its work queue from the SPO list, so the only "config" is
the list itself plus the credential prompts.

## Cloud portability

These were built and proven in **IL6** (sovereign / `microsoft.scloud` endpoints) but port cleanly to commercial or IL5 by swapping host suffixes only — no logic changes.

| Cloud | SPO | Login | Graph (if used) |
|---|---|---|---|
| Commercial | `.sharepoint.com` | `login.microsoftonline.com` | `graph.microsoft.com` |
| IL5 (GCC-H) | `.sharepoint.us` | `login.microsoftonline.us` | `graph.microsoft.us` |
| IL5 (DoD) | `.sharepoint-mil.us` | `login.microsoftonline.us` | `dod-graph.microsoft.us` |
| IL6 (sovereign) | `.spo.microsoft.scloud` | `login.microsoftonline.microsoft.scloud` | `graph.microsoft.scloud` |

SPMT version, module set, AD logic, driver-list schema, and SPO list workflow are identical in every cloud. Cert thumbprints, Entra app IDs, and the `@contoso.gov` UPN suffix are tenant-specific everywhere.

## Why this playbook — On-Prem → SPO (positioning)

### The competitive landscape (2024–2026)

> **Pricing disclaimer.** All vendor cost figures below are **industry
> estimates** based on publicly-discussed federal SI engagements
> 2023–2026. None of these are quoted prices. ShareGate, AvePoint, Quest,
> and BitTitan price by RFQ; ranges vary widely by user count, term, and
> federal premium. **Confirm with a current vendor RFQ before citing any
> number in a customer-facing conversation.**

| Tool | Source coverage | Cloud reach | Licensing | Federal CUI/IL5/IL6 |
|---|---|---|---|---|
| **This playbook (SPMT + scripts)** | SP 2013/2016/2019 sites, MySites | Commercial, GCC, GCC-H/IL5, IL6 | **Free** (Microsoft SPMT) + owned wrapper IP | **Yes — verified in production in IL6** |
| **Microsoft SPMT alone (no wrapper)** | Same SP versions | Commercial, GCC, limited GCC-H | Free | Limited; no orchestration, no AD glue |
| **SharePoint Admin Center bulk migration UI** | SP server, file shares | Commercial, GCC | Free | Limited GCC-H, no IL6 |
| **ShareGate Migrate** | SP 2010–2019, OD on-prem | Commercial, GCC | $30k–$150k+/yr typical | **No IL5/IL6 SaaS instance** |
| **AvePoint Fly + Confidence Platform** | SP 2010–2019, OD, file shares | Commercial, GCC, GCC-H | Six-figure annual | GCC-H yes; IL6 not publicly documented |
| **Quest On Demand Migration (ex-Metalogix Content Matrix)** | SP 2010–2019, OD, file shares | Commercial-hosted SaaS | $50k–$200k+/yr | SaaS routing concerns |
| **BitTitan MigrationWiz** | SP/OD, primarily mailbox-focused | Commercial SaaS | ~$15–$40/user | **No** — SaaS routed through commercial Azure |
| **Tzunami Deployer / Syskit Migrator** | Wide source coverage | Commercial | Mid-tier enterprise | No federal story |

### Why CSAs pick this playbook over the above

1. **Zero licensing cost.** SPMT is free. The wrapper layer (driver list, AD glue, SCA swap, Power Automate hook) is owned IP that ships with the engagement. At federal scale (tens of thousands of users), the licensing delta vs a typical COTS quote is on the order of hundreds of thousands of dollars over a 3-year term — confirm with vendor RFQ for the specific deal.
2. **Sovereign-cloud native.** Same code runs in commercial, GCC-H/IL5, and IL6. Endpoint suffix swap only. Most COTS tools either don't support IL5/IL6 at all (ShareGate, BitTitan, Tzunami) or charge a federal premium for it (AvePoint).
3. **Microsoft-grade fidelity.** SPMT is the same engine Microsoft FastTrack engineers use. No third-party reverse-engineering of SP APIs.
4. **Federal-specific identity workflow** that no COTS tool ships with:
   - AD `wwwHomePage` attribute flip to the new SPO OneDrive URL
   - Removal from `*REDIRECTION*` groups
   - Removal from `SecFltr-USR-OneDrive` and add to `SecFltr-USR-Office365`
   - SCA swap on the new OneDrive: ensure `svc-migration` → make SCA → demote migrated user
   These run by default in the OD2OD script (see the `#<#`/`#>`
   comment-marker caveat in the OD2OD card above — the script's apparent
   block-comment markers do not actually disable the AD code in PowerShell).
5. **No data leaves the tenant.** Source → worker host (customer infra) → SPO. ShareGate Online, BitTitan, Quest On Demand, AvePoint Online all route at least metadata through the vendor cloud — blocked in classified environments.
6. **Auditable PowerShell.** Every behavior is in a `.ps1` file the customer can read, diff in git, and change. COTS tools are opaque binaries.
7. **Concurrency-safe.** The `Inprocess` (OD2OD) / `Processing` (SP2SPO) lock flag lets multiple operators or runners pick from the same list without conflicts. SAC bulk migration doesn't model that.
8. **List-driven control plane.** All state lives in a SharePoint list the customer owns — no external database, no vendor portal. Backups, exports, and offboarding are trivial.
9. **Power Automate notification flow** sits on the list and emails users on `ErrorLog` / `Failed`. Customer owns the messaging; change it without touching the script.

### Honest tradeoffs (say so if asked)

- **No GUI.** Operators must be PowerShell-literate. ShareGate / AvePoint win on operator UX.
- **No pre-migration content classification.** Tools like AvePoint can scan for PII / labels before migration; this playbook doesn't.
- **No vendor SLA.** Internal support model. If a customer needs a vendor to call at 2am, COTS wins.
- **Per-file blocked-extension list (~50) is fixed in `$BlockedExtensions`.** Edit the script header to change.
- **No real-time UI.** The dashboard is on-demand, refreshed by re-running `New-MigrationDashboard.ps1`.

### When NOT to use this playbook

- Cross-tenant migrations (use BitTitan or Microsoft's cross-tenant tooling).
- Mailbox/Teams chat/Yammer content (use Microsoft native or BitTitan).
- Customer has zero PowerShell capability and a hard GUI requirement (consider ShareGate on commercial or AvePoint in GCC-H).
- Project requires pre-migration content classification (consider AvePoint).
